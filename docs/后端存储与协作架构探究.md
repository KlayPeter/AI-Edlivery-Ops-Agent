# AI Delivery Ops Bridge - 后端存储与协作架构评估方案

## 1. 项目的现状
在探讨架构升级前，首先需向研发团队全盘托出当前系统的数据形态、底层运转机制与真实量级，以便大家做出准确的技术评估。

### 1.1 数据模型与形态
目前中台底层完整抽象出了 11 个核心数据实体，从数据形态上可归为以下三类：
- **半结构化/深层嵌套数据（2个）**：
  1. `messages`：飞书 Event 回调的原始深层 JSON。
  2. `contexts`：大模型多轮对话的追踪结构树。
- **强结构化业务数据（6个）**：此类数据字段严谨，强依赖主外键关联（如 `taskId`）。
  3. `tasks`：任务与工单主数据。
  4. `updates`：任务状态变更与操作流水。
  5. `standups`：每日站会打卡记录。
  6. `standup_missing`：每日未打卡人员名单。
  7. `summaries`：大模型生成的每日团队总结报表。
  8. `dashboards`：团队效能看板的快照数据。
- **KV 映射与系统字典（3个）**：
  9. `logs`：系统运行与审计日志。
  10. `idempotency`：基于 event_id 的接口防重校验字典。
  11. `chat_ids`：飞书群组与 TAPD 项目的绑定映射关系。

### 1.2 现有的底层存储机制
当前系统完全剥离了外部数据库，采用**自研的纯本地文件存储（`JsonStore` 类）**，其底层数据落盘机制为：**单记录单文件**。即系统按实体划分文件夹（如 `data/tasks/`），每一条记录的主键 (UUID) 都对应着硬盘上的一个独立的 `.json` 文件。

### 1.3 数据增删改查 (CRUD) 频次特征
基于上述业务与底层机制，当前系统的增删改查操作呈现出极其分化的频次特征：
- **增 (Create) - 高频且并发**：群聊中产生的消息记录 (`messages`) 和接口访问记录 (`logs`)。只要有人在群里互动，就会随时触发这类数据的离散插入，属于最高频的操作。
- **改 (Update) - 中频但易冲突**：工单的状态扭转 (`tasks`) 与站会打卡的更新 (`standups`)。修改频次居中，但目前的实现机制是定位到 `.json` 后进行**全量覆写**。在多用户同时操作一个任务时，极易产生文件锁争抢与数据覆盖。
- **查 (Read) - 相对低频但计算极其昂贵**：分为两类。
  1. 单条查询（高频）：拿 UUID 去读单个文件，速度很快。
  2. 批量/条件查询（低频）：如每日定时“写团队日报”或“渲染效能看板”。虽然每天只执行几次，但在无索引的 JSON 架构下，每一次这类查询都必须调用 `fs.readdirSync` 将成百上千个文件全部读入 Node.js 内存并 `JSON.parse` 反序列化，执行 O(N) 的暴力遍历。
- **删 (Delete) - 极低频**：目前的业务逻辑以追溯和留存为主，几乎不涉及数据的物理删除，仅有少量测试数据的废弃。

### 1.4 真实数据量级预估
按照公司目前 **20-30 人** 的研发团队规模来预估，结合大家日常在企业微信工作群的互动频率，系统每日新增的消息流水、任务状态更新以及系统日志，总计大约在 **500 - 1000 条** 左右。在这个非常“迷你”的量级下，现有的 `JsonStore` 尚能依靠 Node.js 的 V8 引擎在内存中强行进行暴力遍历计算而不至于立刻宕机。

## 2. 现有本地 JSON 存储的风险评估
纯本地 JSON 存储在项目初期的验证阶段，具备极佳的免部署优势（即拉下代码即可直接运行）。但随着业务试用范围的扩大，逐渐暴露出以下底层技术隐患：

- **并发写入冲突**：Node.js 对文件的异步写入机制，在面临多用户高频触发飞书回调时，极易产生文件锁定失败或数据相互覆盖的问题。
- **查询性能瓶颈**：JSON 文件系统缺乏基础的索引支持。对于特定条件过滤（如查询历史延期任务），程序必须执行全量文件的反序列化与内存遍历。当数据规模不断累积时，存在极高的内存溢出与卡顿风险。
- **缺乏一致性保障**：由于不具备 ACID 事务能力，一旦系统在 I/O 写入过程中遭遇容器异常重启，极易产生 JSON 结构断裂，导致核心数据永久性损坏且无法自愈。

## 3. 现阶段是否适合直接上 MySQL？
针对上述的数据一致性与性能隐患，最直接的行业解法肯定是引入 MySQL 或 PostgreSQL 等企业级数据库。但如果在当前试用阶段直接引入，我们可能需要共同探讨并权衡以下几个问题：

- **系统定位是否会导致过度设计？** 目前的AI Delivery Ops Bridge 中台本质上是一个“搬运工”连接器。真正的核心状态在 TAPD，持久聊天记录在飞书。中台本地更多是存“映射关系”和“大模型的临时上下文”。作为一个中间态的缓存层，现阶段就强制绑定一套重量级的数据库底座，会不会太重了？
- **网络通信开销是否划算？** 中台在处理大模型任务时，需要极其频繁地回头去查“上下文”和“防重记录”。如果用 MySQL，每次查询都要走一遍网络握手；而如果是进程内的本地存储，直接在内存和硬盘读写。在目前的架构和并发量下，强行分离数据库所带来的网络延迟是否反而会拖慢响应？

## 4. 探索本地可选的平滑升级方案
如果在现阶段接入 MySQL 确实有些过重，我们是否有既能解决 JSON 并发痛点、又能保持本地轻量化的替代解法？
基于“降低运维门槛”与“保障并发安全”的双重考量，本次调研初步拟定了一个探索方向，供评审探讨：**是否可以考虑引入 SQLite 结合 Prisma ORM？**

如果尝试走这个过渡方案，理论上可能会带来以下改变：
1. **维持零增量运维成本**：SQLite 保持单文件运行，依然无需部署外部服务进程，完美保留现在的开发者体验。
2. **解决并发与查询隐患**：底层拥有 ACID 事务保障高并发写入的排他安全性；自带的 SQL 索引可以把目前的暴力全表扫描优化为极速查询。
3. **保留架构的平滑扩展弹性**：Prisma ORM 能抹平底层方言。这算是一个留给未来的后路：万一将来全量推广导致单机遭遇瓶颈，代码无需重构，只需改一行连接配置就能原地迁移到真实的 MySQL 集群。

## 5. 后期代码和数据维护协助构想
若采用上述本地存储方案（SQLite/JSON），为保障研发团队多人在并行迭代时的效率与安全，拟定以下协同与维护规范草案：

- **代码与数据的环境隔离**：在日常多人开发协作中，个人电脑（开发环境）与线上部署主机（生产环境）必须保持绝对的数据隔离。
  - **后端数据沙盒**：每个人拉取代码后运行生成的本地 `.db` 文件即为各自独立的测试沙盒，绝不与线上真实数据发生任何混用或覆盖。
  - **第三方平台沙盒（飞书与 TAPD）**：为彻底实现环境隔离，每位开发者应当在飞书开发者后台建立自己专属的“测试版机器人”（将长连接指向本地电脑），并在 TAPD 中建立配套的“测试用研发项目”。所有的本地开发调试均应在这套测试沙盒中闭环流转，严禁直连生产环境机器人。
- **仅同步代码与表结构**：团队通过 Git 仅协同业务代码逻辑。对于数据库表字段的新增与修改，由 Prisma 统一生成结构迁移脚本 (`migration`)。线上主机拉取最新代码后，执行迁移命令即可自动完成表结构的无损升级，确保生产数据安全。
- **日常数据运维机制**：
  - **备份策略**：SQLite 单文件特性使得灾备逻辑大幅简化，仅需编写定时任务将 `.db` 文件每日快照拷贝至云盘即可。
  - **数据排查与修复**：告别在繁杂的 JSON 文件夹中进行文本搜索。研发人员可直接使用标准的 GUI 数据库工具（如 DBeaver）直连本地文件，通过标准 SQL 高效完成脏数据的排查与工单状态的应急修复。
