# AI Delivery Ops Bridge 系统架构与业务逻辑详解

这份文档旨在为后续接手维护、二开或扩展本系统的开发者，提供一个清晰的“全局地图”。读完本文档，你将深刻理解这套基于 **TypeScript (Node/Bun)** 的 AI 交付中台是如何运转的。

---

## 1. 系统核心定位

Delivery Ops Bridge 不是一个传统的全量工单系统，而是一个 **连接器与加速器 (Bridge & Accelerator)**。
它的核心职责是：
1. 拦截研发团队在 **飞书 (Feishu)** 中的即时通讯信息。
2. 利用 **大模型 (LLM / DeepSeek)** 理解意图（谁要干嘛、进度如何、有何风险）。
3. 自动同步数据到 **底层系统 (TAPD)**，并将统计和汇总通过 **美观的面板 (Dashboards) / 日报** 反馈给团队。

---

## 2. 整体工程架构

系统采用了彻底的 **前后端分离** 模式：

### 2.1 🚀 后端 (Backend: Bun + Elysia)
- **定位**：机器人的“大脑”与数据中枢。
- **语言**：TypeScript 
- **核心职能**：
  - 暴露高并发 Webhook (`/feishu/webhook`) 供飞书调用。
  - 内置调度引擎 (`Scheduler`) 管理毫秒级的定时任务。
  - 提供 RESTful API 供前端管理台增删改查配置。
  - 通过 `adapters` 模块桥接所有外部系统 (Feishu, TAPD, LLM)。

### 2.2 💻 前端 (Frontend: React + Vite)
- **定位**：管理员的“控制台”。
- **语言**：React (TSX) + Tailwind CSS
- **核心职能**：
  - 解析并渲染后端输出的交付看板。
  - 提供可视化界面，允许 PM/管理员 动态修改组员名单、调整调度时间、配置机器人 Key 等，而无需修改代码或重启服务器。

## 3. 核心业务数据流转逻辑

要维护本项目，你必须熟悉以下三大主干业务流转链路：

### 链路一：群聊建单逻辑 (Task Creation Flow)

当有人在群里发送：“`@AI交付助理 @张三 安排一个支付接口优化的任务，今天下班前搞定，P1优先级`” 时：

1. **Webhook 接收**：飞书推送 HTTP 请求，命中 `backend/src/index.ts` 中的 POST 路由。
2. **事件解析**：`FeishuAdapter.parseEvent` 将飞书的复杂 JSON 转为标准的 `SourceMessage` 对象。
3. **处理器分发**：请求交由 `MessageHandler` (`backend/src/services/messageHandler/index.ts`) 处理。
4. **意图判断**：`TaskParser.parseTaskCommand` 使用正则+关键词判断这是否是一个建单请求。
5. **拆解与同步**：
   - 提取负责人（张三）、Deadline（今天下班前）、优先级（P1）。
   - 调用 `TAPD Adapter` 创建远端 Story。
   - 调用 `JsonStore` 落盘到本地 `data/tasks/`。
   - 调用 `Feishu Adapter` 在群聊回复建单成功的消息，并 @ 相关人。

### 链路二：进度与状态流转逻辑 (Status Update Flow)

当张三在群内回复上一条建单消息：“`我卡住了，因为没权限，@李四 求开权限`” 时：

1. **上下文匹配**：`MessageHandler` 发现这是一条**回复 (Reply)** 消息，会尝试查找被回复的消息是否绑定了某个本地任务 (`Task`)。
2. **大模型意图识别**：因为“我卡住了”、“做完了”这类语言极其灵活，系统会将这句话交给 `MessageIntentParser` (`backend/src/services/messageIntent.ts`)。
3. **AI 推断**：
   - AI 调用 DeepSeek 模型，判定该动作属于 `blocked` (阻塞状态)。
   - 提取原因：“没权限”。
   - 提取协助人：“@李四”。
4. **状态机变更**：
   - `statusUpdates.ts` 捕获到 `blocked` 动作。
   - 更新本地 Task 状态。
   - 通过 `TAPD Adapter` 变更 TAPD 工单状态。
   - 将这笔更新写入 `data/updates/` 方便日后追溯。
   - 机器人发送私聊或群聊消息，正式通知 @李四 需要协助。

### 链路三：全自动定时调度逻辑 (Cron & Summaries)

系统后台有一个 `InProcessScheduler` (`backend/src/core/scheduler.ts`) 实例在循环打点（Tick）：

1. 每分钟读取一次 `config.json` 里的 `schedule` 时间表（例如：`daily_summary: "18:30"`）。
2. 当时间匹配时，触发 `ScheduledJobs` (`backend/src/services/jobs.ts`) 中的对应任务。
3. **生成日报 (Daily Summary)**：
   - 遍历 `data/updates/` 获取今天该群发生的所有状态变更。
   - 打包发给 LLM，要求按固定模板（昨日完成、进度、阻塞风险）生成日报文本。
   - 调用 `Feishu Adapter` 发送到群聊。
4. **生成看板 (Dashboard)**：
   - 统计任务的延期数、各成员的工作负荷。
   - 渲染成静态 HTML。
   - 调用飞书云盘 API 上传 HTML 并回传在线链接。

## 4. 存储层设计：为什么没有 MySQL？

为了做到极致的**轻量化与私有化部署**，系统彻底剥离了重型数据库，全量采用自研的 `JsonStore` (`backend/src/core/storage.ts`)：
- 数据散列存储于 `data/` 目录中。
- 按域划分：`data/tasks/`、`data/standups/`、`data/config/`。
- **优点**：数据可移植性极强，备份只需要打个压缩包。前端修改配置，本质上也是通过 API 更新底层的 JSON 文本。

## 5. 后续扩展与二开指南

如果你是新的维护者，遇到以下需求可以这样修改：

- **需要对接 Jira 或者钉钉？**
  去 `backend/src/adapters/` 下面新增 `jira.ts` 或 `dingtalk.ts`，实现与 `tapd.ts` 或 `feishu.ts` 同样的接口方法，然后在 `HandlerContext` 中完成依赖注入替换。
- **需要修改大模型总结日报的 Prompt？**
  打开 `backend/src/services/summaries.ts`，直接修改传给 `LLMAdapter` 的 System Prompt 即可。
- **需要给管理后台增加一个新页面？**
  前端使用的是标准的 Vite 路由结构。去 `frontend/src/pages/` 下新建 `.tsx` 页面，并在 `App.tsx` 中注册路由。

> **维护原则**：
> 保持核心层 (`core`) 纯净，外部系统的脏数据都在 `adapters` 层完成清洗，所有业务组装都在 `services` 中按职责划分。
