# 项目部署与开发协作流程指南

为了保证 AI Delivery Ops Agent 在多人团队环境下的长期可维护性与稳定性，无论是新增功能还是线上部署，请所有参与开发的成员严格遵循本文档定义的流程。

---

## 一、 项目部署流程 (Deployment Flow)

本项目包含轻量级前端（React/Vite）和后端（Bun/Elysia/SQLite）。我们强烈推荐使用 **Docker** 进行后端部署，以避免不同操作系统下原生 SQLite 编译环境造成的依赖问题。

### 1. 后端容器化部署 (推荐)

**核心要点**：由于我们使用的是 SQLite 本地数据库，**必须挂载持久化数据卷 (Volume)**，否则容器重启将导致所有工单和配置数据丢失。

1. **环境准备**：在宿主机准备一个统一的数据存放目录（例如 `/opt/delivery-ops/data`）以及一份生产环境的 `.env` 文件。
2. **构建镜像**：在项目根目录编写 Dockerfile（可参考 [Bun 官方镜像模板](https://hub.docker.com/r/oven/bun)），执行 `docker build -t delivery-ops-backend .`。
3. **启动容器**：
   ```bash
   docker run -d \
     --name delivery-ops-backend \
     --restart unless-stopped \
     -p 8090:8090 \
     -v /opt/delivery-ops/.env:/app/.env \
     -v /opt/delivery-ops/data:/app/data \
     delivery-ops-backend:latest
   ```
4. **数据备份**：运维人员只需定期通过 `crontab` 备份宿主机的 `/opt/delivery-ops/data/data.db` 即可。

### 2. 前端部署

前端编译后是一堆纯静态资源（HTML/JS/CSS）。
1. 在本地或 CI/CD 机器上运行：
   ```bash
   cd frontend
   pnpm install
   pnpm build
   ```
2. 将 `frontend/dist` 产物丢给 Nginx、Vercel 或飞书微应用托管即可。

## 二、 数据库与多环境数据协作流程 (Data Collaboration)

在多人协同开发时，必须严格遵守我们在 [《后端存储与协作架构探究》](后端存储与协作架构探究.md) 中定下的**“代码与数据环境隔离”**规范：

### 1. 绝对的数据沙盒隔离 (禁止直连生产)
- **后端数据沙盒**：每个人拉取代码后，在本地生成的 `data/data.db` 即为各自独立的测试沙盒数据库。**严禁**索要或拷贝线上真实的 `data.db` 到本地进行开发调试，避免泄露或误污染线上历史记录。
- **第三方平台沙盒**：由于飞书的长连接 (WSClient) 具有排他性，多人不能同时用一个机器人的 Key 连接。每位开发者必须：
  1. 在飞书开发者后台创建一个属于自己的**测试版机器人**（填写自己的 AppID 和 Secret 到本地 `.env`）。
  2. 在 TAPD 建一个自己的测试研发项目。
  3. 所有的本地功能联调，都在自己的沙盒群和测试项目中闭环流转。

### 2. 我们如何共享和修改数据结构？
我们在团队间**只共享代码和表结构 (Schema)，绝不共享业务数据**。

当业务需要修改表结构（如给 Task 表增加 `is_urgent` 字段）时：
1. **修改 Schema**：打开 `backend/prisma/schema.prisma` 添加新字段。
2. **生成迁移脚本**：运行 `bunx prisma migrate dev --name add_urgent_field`。
3. **提交代码**：此时会生成一个 `prisma/migrations/2026xxxx_add_urgent_field` 的目录。将这个目录以及 `schema.prisma` 一起用 Git 提交推送到远端。

### 3. 同步同事的数据库更新
当你从 Git 拉取了别的同事提交的代码后，如果发现有新的 Migrations 文件夹：
1. **执行迁移**：运行 `bunx prisma migrate dev`。
2. Prisma 会自动对比你本地的 SQLite 结构，并顺序执行缺失的 SQL 脚本，保持你的本地数据库结构与团队同步，但**不会影响你的本地测试数据**。

> **⚠️ 严禁使用 `bun run db:push`**：在多人协作项目中，`db push` 会强制覆盖并丢失部分字段。只能在彻底清空重做本地数据库的极少数情况下使用。

## 三、 后端代码开发协作流程 (Code Collaboration)

### 1. 架构约定与 Handler 分离
为了避免再次出现代码长达数百行的“石山 (God Object)”，我们在 `backend/src/services/messageHandler/` 中引入了基于意图的“策略路由”模式。

- **新增飞书指令**：如果你想给机器人加一个新功能（例如：“@机器人 请假”）：
  1. **禁止**在 `index.ts` 里直接堆叠 `if-else`。
  2. 在 `handlers/` 目录下新建一个专门的 `LeaveHandler.ts`。
  3. 实现统一的 `maybeHandle(ctx, message)` 静态方法。
  4. 回到 `index.ts`，将你的 `LeaveHandler.maybeHandle` 加入到 handlers 调度数组中。

### 2. 避免内存状态陷阱
- Node.js / Bun 是单线程异步架构。所有涉及到业务状态变更的数据，必须直接通过 `PrismaStore` 读写，**严禁在代码里使用全局数组或 Map 来长期缓存工单数据**，这在 WebSocket 高并发推送时会导致极其严重的数据读写冲突。

### 3. 日常测试要求
由于本系统的长连接事件受限于飞书的真实推送，请遵循以下开发步骤：
1. **类型检查**：任何 TypeScript 逻辑改动，提交代码前必须运行 `bun run typecheck` 保证通过。
2. **单元测试**：针对大模型的 Prompt 提取逻辑、文本清洗正则逻辑，应尽量提取为独立函数并使用 `bun test` 编写覆盖。
3. **沙盒群联调**：上线前必须在一个测试群中触发全量指令流转，确保没有 Regression。

## 四、 环境变量与安全 (Env Security)

- 所有的 API Key（飞书的 App ID/Secret，TAPD 的账号，大模型的 API Key）全部放置在 `backend/.env`。
- `.env` 严禁提交进 Git！
- 每次代码里读取了新的环境变量，**必须**同步更新 `backend/.env.example`，并留下人类可读的注释（说明去哪里申请这个 Key）。
