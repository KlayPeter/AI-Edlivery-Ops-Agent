# Delivery Ops Bridge - 后端存储与协作架构设计方案

这是一份写给后端团队的存储方案及协作设计报告。我们在保证专业架构思考的同时，尽量用大白话把“为什么换”、“换成什么”、“多人怎么协作”以及“后续怎么维护”说清楚。

---

## 1. 我们的现状是什么样子的？
目前，我们的 AI 交付中台完全依赖 **本地 JSON 文件** 来存数据。
比如，一条飞书消息或者一个任务生成后，后端会直接在 `data/` 文件夹里创建一个对应的 `.json` 文件写进去。整个系统非常轻量，根本不需要连什么数据库，代码跑起来就能用。

## 2. 觉得有什么问题？（痛点在哪里）
这种“写本地文件”的做法，刚起步做 Demo 时没问题，但要真正在公司推广，会遇到四个致命伤：

1. **多人同时用的“撞车”问题（无并发控制）**
   当群里有两三个人同时 `@机器人` 时，程序就会同时去写同一个 JSON 文件。这极容易导致文件内容被互相覆盖，甚至写乱码，导致整个系统崩溃。
2. **查个数据能把服务器卡死（无索引查询慢）**
   我们马上要做前端报表了。比如你想查“张三上周所有延期的紧急任务”，程序只能把几千个 JSON 文件全部打开、读进内存、一个个挑。数据一多，服务器内存直接撑爆。
3. **断电或者重启会丢数据（无事务保护）**
   如果我们正在往文件里写数据，刚写了一半服务器突然重启了，这个 JSON 文件就彻底损坏了（比如少了个右括号），没有回滚和自愈的能力。
4. **没办法多台机器一起跑（无法扩展）**
   因为数据都死死绑在这一台电脑的硬盘上，以后如果我们想加一台服务器分担流量，两台机器根本没法共享数据。

## 3. 需要哪方面的提升？
基于上面的坑，我们需要一个新的存储方案，它必须满足：
- **安全不丢数据**：别人同时发消息不能乱，断电重启不能坏。
- **查询要快**：能建立索引，支持复杂的报表数据拉取，不能动不动就全表扫描。
- **千万不要太折腾**：这是最关键的！我们做的是个轻量级中台，千万不要为了存个日志，就非逼着公司去采购或搭建一整套沉重的数据库集群。

## 4. 目前计划升级的方案是哪个？有什么用？
**目前计划升级的方案是：SQLite 数据库 + Prisma (代码查询工具)。**

**为什么选它？**
SQLite 是一个神奇的数据库，它**不用安装任何服务端软件**，整个数据库依然只是一个本地的 `.db` 文件。

**它解决了什么问题？**
- **解决了撞车和损坏问题**：它底层支持完整的数据库事务（ACID）。哪怕 100 个人同时发消息，它也会排好队一个个安全写进去；哪怕中途拔电源，数据也绝对不会坏。
- **解决了查询慢的问题**：它支持标准的 SQL 语法。我们可以给时间、人员打上“索引”，查询速度比读 JSON 文件快上百倍，内存占用极其微小。
- **保留了“轻量”的初心**：运维成本依然是 0。别人拷贝我们的项目，依然是一键启动，不用去配置繁琐的账号密码和网络白名单。

## 5. 还在评估中的架构储备（其他备选方案）
对于业界主流的几种重量级存储方案，我们也都在同步评估中。考虑到当前业务阶段的核心诉求是“轻量推广”，我们将它们作为中长期发展的架构储备选项，视业务规模的变化灵活选用：

- **备选方案 A：MySQL / PostgreSQL**
  - **优点**：真正的企业级霸主，支持多机横向扩展与分布式高可用，数据共享能力极强。
  - **当前考量**：由于需要专门的服务器资源来进行独立部署和日常维护，暂时显得比我们现阶段“开箱即用”的轻量化定位略重一些。但我们的代码底层已经引入了 Prisma 工具，未来一旦需要跨部门大规模部署，改一行配置即可直接无缝升级为 MySQL/PG 集群。
- **备选方案 B：MongoDB**
  - **优点**：非结构化的文档存储，非常契合飞书复杂的 JSON 事件以及大模型多变的对话上下文。
  - **当前考量**：在生成前端面板所需的“多表复杂关联统计”（比如组员效能聚合报表）时，关系型数据库往往表现更佳。后续如果大模型分析历史记录的需求爆发，我们会考虑将其引入作为专项存储。
- **备选方案 C：Redis**
  - **优点**：极速的内存操作，天然适合做高频事件的防重去重和接口限流。
  - **当前考量**：后续随着群聊并发量增大，我们非常倾向于引入它来作为核心数据库（如 SQLite）的“外挂加速器”，用来抗压和做临时缓存，目前暂且不把它作为长期任务履历的主存储库。

## 6. 团队多人协作时，数据该如何同步？
对于像我们这种采用本地存储（无论 JSON 还是 SQLite）的项目，当多名开发人员一起迭代时，**数据应该如何共享和同步？**

### 核心原则：数据绝对隔离，各搞各的沙盒
在真实研发协作中，**你个人的电脑、同事的电脑、以及线上正式部署的主机**，绝对不应该共享同一份业务数据！

- **线上主机（生产环境）**：里面存的是全公司真正干活的业务数据，这部分数据是最宝贵的。
- **个人电脑（开发环境）**：大家拷贝下来的项目，运行生成的都是“独立沙盒”。你在本地发给机器人的测试消息，只会保存在你自己电脑的 `.db` 或 `.json` 里，绝不会污染线上主机的真实数据。建议每个开发者在飞书后台建立一个“测试版机器人”专门用于自己本地调试。

### 我们真正要同步的是什么？
我们只需要通过 Git 同步两样东西：**代码逻辑** 和 **表结构**。
1. **代码合并**：你在本地开发完新功能后，把代码提交到 Git 仓库。
2. **结构自动升级**：如果我们采用 SQLite + Prisma，线上主机拉取最新代码后，只需执行一行命令（如 `npx prisma migrate`）。线上服务器就会聪明地在保留原有真实数据的基础上，自动为它的本地数据库添加上新字段。

所以，本地数据的特性反而成了一个优势：**让每个开发者都能零成本拥有一个绝对安全的独立测试沙盒**。

## 7. 后续运维：数据该怎么维护？
升级到 SQLite 之后，日常的业务数据维护会变得非常轻松规范：

1. **日常数据备份极其简单**
   SQLite 依然是一个单纯的文件（比如 `data/production.db`）。要备份全公司半年的研发履历，只需要写一个定时脚本，每天凌晨把这个文件拷贝一份到云盘或者对象存储即可，比备份 MySQL 简单得多。
2. **表结构的安全变更**
   如果产品经理想在“任务单”里加一个“预期延期天数”的字段。开发人员只需在 `schema.prisma` 文件里写上这个字段，通过 Prisma 生成一套“变更脚本”。发布上线时自动执行脚本即可，全程无痛，且能随时回滚。
3. **线上脏数据的排查与修复**
   比起在一堆 JSON 文件里找数据，有了 SQLite 后，你可以使用任何通用的可视化数据库工具（比如 DBeaver、Navicat 甚至 VSCode 里的插件），直接打开这个 `.db` 文件。想查谁发的消息、想修改某条搞错了的工单状态，直接写一句 SQL 或者在可视化界面里点两下就改好了。
